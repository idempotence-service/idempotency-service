# Требования к проекту

Документ: Требования к подсистеме идемпотентной обработки событий для микросервисной архитектуры

Версия: 1.0

Дата: 20.03.2026

## 1. Общие сведения

### 1.1. Наименование системы

Подсистема идемпотентной обработки событий и защиты от повторного применения побочного эффекта.

### 1.2. Назначение системы

Система предназначена для обеспечения правила "одна бизнес-операция - один эффект" при повторной отправке сообщений, клиентских ретраях, таймаутах, сетевых сбоях и повторной доставке событий между внешней системой-отправителем, ядром обработки и внешней системой-получателем.

### 1.3. Область действия документа

Настоящий документ устанавливает функциональные и нефункциональные требования к:

- ядру подсистемы идемпотентности;
- механизму хранения и обработки статусов выполнения;
- механизму технических ответов отправителю;
- механизму доставки уникальных событий получателю;
- API для ручного разбора ошибок;
- вспомогательному локальному стенду для тестирования и приемки.

### 1.4. Границы проекта

В текущий объем работ входят:

- подсистема идемпотентной обработки событий;
- локальный тестовый стенд с отправителем и получателем;
- конфигурация маршрутов через YAML;
- запуск через Docker Compose.

В текущий объем работ не входит реализация собственной промышленной очереди сообщений, log compaction, consumer groups собственного брокера и функциональность отдельной административной панели.

## 2. Термины и определения

- Идемпотентность - свойство операции, при котором повторное выполнение с тем же идентификатором не приводит к повторному бизнес-эффекту.
- Idempotency key - уникальный идентификатор бизнес-операции.
- Global key - внутренний ключ системы, формируемый на основе сервиса, интеграции и внешнего идентификатора сообщения.
- Побочный эффект - внешнее наблюдаемое действие, например отправка события получателю.
- Outbox - таблица или очередь внутренних записей для надежной отправки результата обработки во внешнюю систему.
- Receipt - зафиксированный результат ранее обработанной операции.
- Ручной разбор - сценарий просмотра и повторного запуска задач, завершившихся ошибкой.
- Отправитель - внешняя система, передающая события в ядро.
- Получатель - внешняя система, принимающая уникальные события из ядра и возвращающая результат обработки.

## 3. Контекст системы

Система должна работать в следующем логическом контуре:

1. Внешняя система-отправитель публикует событие во входной Kafka-совместимый брокер.
2. Ядро системы принимает событие, проверяет его на идемпотентность и сохраняет состояние обработки.
3. Для первого уникального события ядро отправляет получателю уникальное сообщение.
4. Для повторных событий ядро не должно повторно инициировать бизнес-эффект.
5. Отправителю возвращается технический результат обработки.
6. Получатель может вернуть асинхронный ответ об успехе, ошибке или необходимости повторной отправки.
7. При ошибках должна быть доступна процедура ручного разбора через HTTP API.

## 4. Внешние интерфейсы

### 4.1. Интерфейс конфигурации

Система должна использовать YAML-файл маршрутов, содержащий:

- данные о сервисе;
- описание интеграций;
- параметры Kafka producer и consumer для стороны отправителя;
- параметры Kafka producer и consumer для стороны получателя;
- признак включения идемпотентности.

### 4.2. HTTP API ядра

Система должна предоставлять HTTP API для ручного разбора:

- получение списка ошибочных задач;
- получение детальной информации по событию;
- повторный запуск задачи.

### 4.3. HTTP API тестового стенда

Для целей тестирования система должна предоставлять:

- API отправителя для отправки событий и просмотра технических ответов;
- API получателя для просмотра доставленных событий, переключения режима обработки и ручной отправки ответа.

### 4.4. Kafka-интерфейсы

Система должна поддерживать:

- прием входящих событий от отправителя;
- отправку технических ответов отправителю;
- отправку уникальных событий получателю;
- прием асинхронных ответов от получателя.

Формат сообщений должен быть JSON и содержать заголовки сообщения и полезную нагрузку.

## 5. Функциональные требования

| ID | Требование |
| --- | --- |
| FR-01 | Система должна принимать входящие события из Kafka-совместимого брокера в соответствии с маршрутами, заданными в YAML-конфигурации. |
| FR-02 | Система должна валидировать входящее событие. В событии обязательно должны присутствовать идентификатор `uid` и полезная нагрузка. |
| FR-03 | Система должна формировать внутренний `global key` по правилу `<service>:<integration>:<uid>`. |
| FR-04 | Система должна сохранять состояние обработки события в распределенно доступном постоянном хранилище. |
| FR-05 | Система должна поддерживать статусы обработки не менее чем: `RESERVED`, `WAITING_ASYNC_RESPONSE`, `COMMITTED`, `ERROR`. |
| FR-06 | При первом получении уникального события система должна создать запись идемпотентности и зафиксировать факт успешного прохождения проверки на дубликат. |
| FR-07 | При повторном получении события с тем же `global key` система не должна повторно инициировать внешний бизнес-эффект. |
| FR-08 | Система должна обеспечивать корректную работу при конкурентной обработке нескольких запросов с одним и тем же `global key`. |
| FR-09 | Для первого уникального события система должна формировать технический ответ отправителю со статусом успешного прохождения проверки. |
| FR-10 | Для повторного события система должна формировать технический ответ отправителю со статусом отказа по причине дублирования. |
| FR-11 | Технические ответы отправителю должны отправляться через механизм outbox, обеспечивающий отделение сохранения результата от фактической отправки в Kafka. |
| FR-12 | Для первого уникального события система должна отправлять получателю только одно уникальное сообщение по маршруту, указанному в конфигурации. |
| FR-13 | Если для маршрута настроен асинхронный ответ получателя, система должна переводить задачу в статус `WAITING_ASYNC_RESPONSE` до получения ответа. |
| FR-14 | При получении ответа получателя с результатом `SUCCESS` система должна переводить задачу в статус `COMMITTED`. |
| FR-15 | При получении ответа получателя с результатом `FAIL` и признаком повторной отправки система должна переводить задачу обратно в статус `RESERVED`. |
| FR-16 | При получении ответа получателя с результатом `FAIL` без признака повторной отправки система должна переводить задачу в статус `ERROR`. |
| FR-17 | Система должна сохранять аудит обработки некорректных сообщений, дублей, технических ошибок и асинхронных ответов без ожидающей операции. |
| FR-18 | Система должна предоставлять HTTP-метод для получения списка задач со статусом `ERROR` с поддержкой пагинации и сортировки. |
| FR-19 | Система должна предоставлять HTTP-метод для получения карточки события по `global key`. |
| FR-20 | Система должна предоставлять HTTP-метод для повторного запуска задачи со статусом `ERROR`. |
| FR-21 | При повторном запуске задачи через API система должна переводить ее в статус `RESERVED` и возвращать задачу в общий поток обработки. |
| FR-22 | Система должна загружать YAML-конфигурацию маршрутов при старте приложения и отклонять запуск при некорректной конфигурации. |
| FR-23 | Система должна выполнять периодическую очистку успешно завершенных записей, срок хранения которых превысил заданное значение retention. |
| FR-24 | Система должна поддерживать настраиваемый срок хранения успешно завершенных записей. |
| FR-25 | Система должна предоставлять endpoint проверки работоспособности сервиса. |
| FR-26 | Для целей приемки система должна включать сервис-имитацию отправителя, позволяющий отправлять обычные события и дубликаты, а также получать технические ответы. |
| FR-27 | Для целей приемки система должна включать сервис-имитацию получателя, позволяющий принимать события, фиксировать повторные доставки, переключать режимы обработки и отправлять ручные ответы. |
| FR-28 | Система должна запускаться локально в виде единого стенда через Docker Compose. |

## 6. Нефункциональные требования

| ID | Требование |
| --- | --- |
| NFR-01 | Система должна обеспечивать корректность правила "одна бизнес-операция - один внешний эффект" для всех повторных сообщений с одинаковым `global key`. |
| NFR-02 | Система должна сохранять состояние обработки и не терять его при перезапуске экземпляра сервиса. |
| NFR-03 | Система должна обеспечивать возможность запуска не менее двух экземпляров ядра на общем хранилище без нарушения идемпотентности. |
| NFR-04 | На тестовом стенде система должна обеспечивать устойчивую обработку не менее 20 входящих событий в секунду при размере полезной нагрузки до 64 КБ без потери корректности результата. |
| NFR-05 | Для HTTP API ручного разбора время ответа для 95% запросов не должно превышать 2 секунд при одновременной работе не менее 10 пользователей тестового стенда. |
| NFR-06 | Все временные значения в хранилище и логах должны фиксироваться в UTC. |
| NFR-07 | Конфигурация должна задаваться внешними параметрами и файлами без необходимости пересборки приложения при смене маршрутов или параметров окружения. |
| NFR-08 | Ручки ручного разбора должны быть защищены аутентификацией по bearer token и ролевой авторизацией. |
| NFR-09 | Система не должна хранить секреты и учетные данные в исходном коде репозитория. |
| NFR-10 | Система должна формировать журнал событий и аудита, достаточный для разбора ошибок, дублей и ручных перезапусков. |
| NFR-11 | Система должна быть развертываемой на Linux-окружении с использованием Docker и Docker Compose. |
| NFR-12 | Система должна использовать Java 21, PostgreSQL 16 и Kafka-совместимый брокер сообщений. |
| NFR-13 | Система должна быть модульной: логика ядра, общий инфраструктурный слой, имитатор отправителя и имитатор получателя должны быть разделены на самостоятельные модули. |
| NFR-14 | Система должна сопровождаться документацией, достаточной для локального запуска, ручной проверки и базовой эксплуатации стенда. |
| NFR-15 | Проект должен содержать автоматизированные интеграционные тесты, покрывающие как минимум сценарии дублирования, повторной отправки, ручного перезапуска, защиты API и очистки данных. |

## 7. Требования к данным

### 7.1. Данные состояния обработки

Для каждой бизнес-операции должны храниться не менее чем следующие данные:

- `global key`;
- исходный внешний идентификатор `uid`;
- имя сервиса;
- имя интеграции;
- снимок маршрута из конфигурации;
- заголовки сообщения;
- полезная нагрузка;
- текущий статус;
- описание статуса или ошибки;
- дата и время создания;
- дата и время последнего обновления.

### 7.2. Данные outbox

Для каждой записи outbox должны храниться не менее чем:

- ссылка на `global key`;
- маршрут отправки;
- технический результат обработки;
- статус отправки outbox-записи;
- дата и время создания;
- дата и время последнего обновления.

### 7.3. Данные аудита

Для каждой аудиторской записи должны храниться не менее чем:

- идентификатор события, если он определен;
- сервис и интеграция;
- причина записи в аудит;
- исходные заголовки, полезная нагрузка и снимок маршрута при наличии;
- дата и время создания записи.

## 8. Требования к режимам эксплуатации

### 8.1. Режим запуска

Система должна поддерживать:

- локальный запуск на машине разработчика;
- запуск всех компонентов через Docker Compose;
- раздельный запуск модулей в среде разработки.

### 8.2. Режим тестирования

Система должна поддерживать тестовые сценарии:

- отправка одиночного события;
- отправка дубликатов;
- ошибка обработки на стороне получателя;
- повторная отправка по запросу получателя;
- ручной перезапуск ошибочной задачи;
- проверка защищенности API.

## 9. Критерии приемки

Проект считается соответствующим требованиям при выполнении всех условий:

1. Локальный стенд поднимается штатно через Docker Compose.
2. Первое событие с новым `uid` проходит полный путь до получателя и завершается статусом `COMMITTED`.
3. Повторное событие с тем же `uid` не приводит к повторной доставке уникального события получателю.
4. Отправитель получает технический результат обработки как для первого события, так и для дубликата.
5. Получатель может вернуть успешный ответ, ошибку без повтора и ошибку с повторной отправкой.
6. Ошибочная задача доступна через API ручного разбора и может быть повторно запущена.
7. Защищенные ручки недоступны без корректного bearer token.
8. Старые успешно завершенные записи удаляются по политике retention.
9. Автоматизированные интеграционные тесты проходят успешно.

## 10. Ограничения и допущения

1. Система реализуется как учебный проект и ориентирована на демонстрацию корректной семантики идемпотентной обработки, а не на промышленный уровень отказоустойчивости кластера.
2. В качестве брокера сообщений допускается использование Kafka-совместимого решения.
3. Аутентификация внутренних Kafka-взаимодействий в рамках локального стенда может выполняться без TLS и SASL.
4. Текущий объем проекта не требует реализации собственного пользовательского интерфейса и собственной очереди сообщений.
